# 🚀 Инструкция по развертыванию на тестовом и продовом серверах

## Архитектура

- **Тестовый сервер**: домен с `test` в названии → ветка `test`
- **Продовый сервер**: основной домен → ветка `main`

Оба сервера используют один Git репозиторий с разными ветками.

## Автоматическое определение окружения

Скрипты `deploy.sh` и `update.sh` автоматически определяют окружение по домену в конфигурации nginx (`nginx/conf.d/rit-utils.conf`):

- Если домен содержит `test` → используется ветка `test`
- Если домен основной (без `test`) → используется ветка `main`

## Настройка тестового сервера

### 1. Первичное развертывание

```bash
cd /path/to/rit-utils

# Убедитесь, что в nginx/conf.d/rit-utils.conf указан тестовый домен (содержит "test")
# Скрипт автоматически определит окружение и использует ветку test

./deploy.sh
```

### 2. Обновление после изменений в ветке test

```bash
cd /path/to/rit-utils

# Скрипт автоматически определит, что это тестовый сервер (по домену)
./update.sh

# Или явно указать ветку
./update.sh test
```

## Типичный workflow

### 1. Разработка и тестирование

```bash
# На локальной машине или в CI/CD
git checkout test
# ... вносите изменения ...
git add .
git commit -m "Новая фича"
git push origin test
```

### 2. Развертывание на тестовый сервер

```bash
# На тестовом сервере
cd /path/to/rit-utils
./update.sh

# Скрипт автоматически:
# - Определит, что это тестовый сервер (по домену в nginx конфиге)
# - Переключится на ветку test
# - Получит изменения из Git
# - Перезапустит контейнеры
# - Проверит доступность приложения
```

### 3. После успешного тестирования - деплой в прод

```bash
# На локальной машине
git checkout main
git merge test
git push origin main
```

### 4. Развертывание на продовый сервер

```bash
# На продовом сервере
cd /path/to/rit-utils
./update.sh

# Скрипт автоматически:
# - Определит, что это продовый сервер (по домену в nginx конфиге)
# - Переключится на ветку main
# - Получит изменения из Git
# - Перезапустит контейнеры
# - Проверит доступность приложения
```

## Принудительное указание ветки

Если нужно принудительно указать ветку (например, для тестирования):

```bash
# На тестовом сервере - использовать ветку main (не рекомендуется)
./update.sh main

# На продовом сервере - использовать ветку test (не рекомендуется)
./update.sh test
```

## Переменные окружения

Можно также использовать переменную окружения `ENV`:

```bash
# Тестовое окружение
ENV=test ./deploy.sh
ENV=test ./update.sh

# Продовое окружение
ENV=prod ./deploy.sh
ENV=prod ./update.sh
```

## Проверка текущего окружения

Скрипты автоматически выводят информацию об окружении:

```
🔄 Обновление RIT-Utils
======================
ℹ️  Окружение: 🧪 ТЕСТОВОЕ
ℹ️  Целевая ветка: test
ℹ️  Домен: your-test-domain.com
```

или

```
🔄 Обновление RIT-Utils
======================
ℹ️  Окружение: 🚀 ПРОДОВОЕ
ℹ️  Целевая ветка: main
ℹ️  Домен: your-production-domain.com
```

## Важные замечания

1. **Конфигурация nginx**: Убедитесь, что в `nginx/conf.d/rit-utils.conf` указан правильный домен для каждого сервера
2. **SSL сертификаты**: Каждый сервер должен иметь свои SSL сертификаты для своего домена
3. **Git репозиторий**: Оба сервера должны иметь доступ к одному и тому же Git репозиторию
4. **Ветки**: Не смешивайте ветки между серверами без необходимости

## Устранение неполадок

### Скрипт не определяет правильное окружение

Проверьте конфигурацию nginx:
```bash
grep "server_name" nginx/conf.d/rit-utils.conf
```

Должно быть:
- На тестовом сервере: `server_name your-test-domain.com ...` (домен содержит "test")
- На продовом сервере: `server_name your-production-domain.com ...` (основной домен)

### Неправильная ветка

Принудительно укажите ветку:
```bash
./update.sh test  # для тестового сервера
./update.sh main  # для продового сервера
```

### Проблемы с доступом к Git

Убедитесь, что на сервере настроен доступ к Git репозиторию:
```bash
git remote -v
git fetch origin
```
